# Phase 7: Validate — ユースケース・シナリオ・UI ウォークスルーによるモデル検証

## 目的

設計したドメインモデルが実際のユースケースや画面操作で破綻しないかを対話的に検証する。
DDD Distilled Ch.2 の「Putting Scenarios to Work」と Ch.7 の「Timeboxed Modeling」に基づく。

## 3つの検証アプローチ

### A. シナリオウォークスルー

ビジネスシナリオを自然言語で記述し、モデルがそのシナリオを表現できるか逐次確認する。

DDD Distilled の手法:
- ユビキタス言語でシナリオを書く (ユーザーストーリーやユースケースではなく「ドメインシナリオ」)
- 「誰が」「何を」「どの条件で」「どうなるか」を明確にする
- 制約条件 (ガード条件) を洗い出す

### B. UI ウォークスルー

画面遷移を追いながら、各操作がドメインモデルのどの Command / Aggregate / Event に対応するかを検証する。

### C. Given/When/Then (BDD スタイル)

実行可能な仕様として記述し、受入テストの基盤にする。

## 進行手順

### Step 1: 検証対象の選定

質問:
- 「最も重要なユースケースは何ですか? ユーザーが毎日行う操作は?」
- 「最も複雑な操作、エッジケースが多い操作は?」
- 「既存の画面やモックアップはありますか?」

優先度の判断基準:
1. Core Domain のユースケースを最優先
2. 複数の Aggregate を横断する操作
3. 複数の Bounded Context にまたがるフロー

### Step 2: シナリオの記述

ユーザーと一緒にシナリオを書く。1シナリオずつ。

形式:
```
シナリオ: {シナリオ名}
  {アクターが}{操作を行う}
  {前提条件: 〇〇の場合のみ}
  {ガード条件: △△でなければ〜}
  {結果: □□になる、☆☆に通知される}
```

各文を書くたびに:
- 「この用語はユビキタス言語に含まれていますか?」
- 「このアクターはどのロールですか?」
- 「この前提条件は誰が保証しますか? どの Aggregate の不変条件ですか?」

### Step 3: モデルとの照合

シナリオの各ステップを、ドメインモデルの構成要素にマッピング:

| シナリオのステップ | Command | Aggregate | Event | 備考 |
|---------------|---------|-----------|-------|------|
| {操作の記述} | {対応する Command} | {対応する Aggregate} | {発行される Event} | {見つかった問題} |

問題が見つかるパターン:
- **Command が存在しない** → Aggregate に操作が足りない
- **Aggregate が特定できない** → モデルに概念が欠落
- **Event が必要だが定義されていない** → イベント設計に漏れ
- **複数の Aggregate を同時に更新する必要がある** → トランザクション境界の再検討
- **UI の表示に必要な情報が取得できない** → Read Model / Query の設計不足

### Step 4: UI ウォークスルー (画面がある場合)

画面単位で進める:

```
画面: {画面名}
  表示データ:
    - {フィールド} ← {どの Aggregate / Query から取得}
  操作:
    - {ボタン/アクション} → Command: {名前} → Aggregate: {名前}
  遷移:
    - {条件} → {次の画面}
```

各画面について:
- 「この画面に表示する情報は、どの Aggregate から取得しますか?」
- 「複数の Aggregate のデータを合成する必要がありますか? → Read Model が必要?」
- 「このボタンを押したとき、裏で何が起きますか?」

### Step 5: Given/When/Then の記述 (任意)

BDD スタイルで実行可能な仕様にする:

```
Scenario: {シナリオ名}
  Given {前提条件}
  And {追加の前提}
  When {操作}
  Then {期待する結果}
  And {追加の結果}
```

Red-Green アプローチ:
1. 仕様を書く (Red — まだ通らない)
2. ドメインモデルを実装して仕様を通す (Green)
3. リファクタリング

### Step 6: 発見の記録とフィードバック

検証で見つかった問題を他のフェーズにフィードバック:
- Aggregate の不変条件の漏れ → Phase 5 (aggregates) へ
- 足りない Domain Event → Phase 6 (events) へ
- 用語の不整合 → Phase 8 (glossary) へ
- Bounded Context の境界問題 → Phase 3 (contexts) へ

## --analyze 時の追加手順

1. 既存のコントローラー / API エンドポイントを読む
2. 既存のテストコード (特に受入テスト) を読む
3. API エンドポイントとドメインモデルの対応を自動マッピング
4. テストされていないユースケースを指摘

## --challenge 時の追加質問

- 「このシナリオの happy path はわかりましたが、異常系は? ユーザーが途中で中断したら?」
- 「この画面の操作を同時に2人が行ったらどうなりますか?」
- 「このデータが存在しない場合、画面はどう振る舞うべきですか?」
- 「この操作の権限がないユーザーがアクセスしたら?」
