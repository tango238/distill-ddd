# Phase 1: Discover — Core Domain の特定

## 目的

ビジネスの全体像を理解し、Core Domain (戦略的に最も重要な領域) を特定する。
Problem Space と Solution Space を区別し、DDD を適用すべき範囲を決める。

## 進行手順

### Step 1: ビジネス概要のヒアリング

質問例:
- 「このソフトウェアが解決するビジネス上の問題は何ですか?」
- 「このシステムの顧客は誰で、何を達成したいですか?」
- 「競合と比較して、このシステムの差別化ポイントは?」

### Step 2: Core Domain の仮特定

質問例:
- 「このビジネスで最も重要な機能は何ですか? それがなければ事業が成り立たないものは?」
- 「最も優秀なエンジニアを投入すべき領域はどこですか?」
- 「この領域は自社開発すべきか、既製品で代替可能か?」

### Step 3: サブドメインの分類

ユーザーの回答から以下の3種類に分類:

| 種別 | 説明 | 投資レベル |
|------|------|---------|
| **Core Domain** | 競争優位の源泉。最大の投資 | Ubiquitous Language を精緻に開発 |
| **Supporting Subdomain** | Core を支えるが差別化要因ではない。既製品なし | カスタム開発するが Core ほど投資しない |
| **Generic Subdomain** | 汎用的。既製品で代替可能 | 購入 or 最小限の開発 |

### Step 4: SWOT 分析 (任意)

DDD Distilled Ch.7 の SWOT フレームワーク:
- **Strengths**: プロジェクトの強み (ドメイン知識、技術力)
- **Weaknesses**: 改善が必要な領域 (Big Ball of Mud、知識のギャップ)
- **Opportunities**: 活用できる要素 (新しい市場、技術的優位)
- **Threats**: リスク要因 (レガシーシステム、競合)

## --analyze 時の追加手順

1. プロジェクトの README、CLAUDE.md、docs/ を読む
2. モジュール構成からサブドメイン候補を推測
3. 「現在のコードベースから推測すると、Core Domain は〇〇に見えます。合っていますか?」
